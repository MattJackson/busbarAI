import { Tabs, TabItem } from '@astrojs/starlight/components';

The only honest way to answer "how much does Busbar add?" is to **change exactly one thing**. So this
benchmark fixes everything — the machine, the CPU cores, the request count, the concurrency, the client,
the binary — and varies **only the upstream Busbar talks to**: first an *instant local mock* (to isolate
Busbar's own cost), then the *real Anthropic API* (to show what that cost looks like in the wild). Same
rig, same load, one variable. Read **Result** for the numbers; **Reproduce** to run the identical thing
yourself.

<Tabs>
  <TabItem label="Result">
    ## The setup (identical for every row below)

    One AWS **`c7g.2xlarge`** (8 vCPU, Graviton3) in **us-east-1**, Amazon Linux 2023, running the
    released **`v1.0.0-rc.5`** `aarch64` binary. To make the number Busbar's and nothing else's:

    - **Cores are pinned.** Busbar is `taskset`-bound to a fixed set of cores; the load generator and the
      mock get *different* cores. So Busbar is never starved by the test harness, and its core budget is
      fixed and disclosed — it cannot "cheat" by spreading across the whole box.
    - **Busbar's CPU is logged** (from `/proc`) on every run, so you can see it uses **under one core** —
      the low latency is efficiency, not core-hogging.
    - **The load generator is `oha`** (Rust, low-jitter), percentiles captured as **JSON** (no fragile
      text parsing). Each cell is **`__N__` requests, repeated `__R__×`** — every percentile, including
      p99.9, is drawn from **__N__+ samples**.
    - **The delta is a subtraction.** For each upstream we measure two paths against *that same upstream*
      and subtract per percentile: `client → upstream` (baseline) vs `client → Busbar → upstream`. What's
      left is Busbar.

    The **only** thing that differs between the two results below is the upstream URL.

    ## Result 1 — vs an instant mock: Busbar's own cost

    The mock returns a fixed response in microseconds with zero jitter, so the subtraction is *pure
    Busbar*. We run it two ways — a same-protocol passthrough and a full cross-protocol translation —
    because translation is the work that matters.

    | Path (vs instant mock) | Δ p50 | Δ p99 | Δ p99.9 | Busbar CPU |
    |---|---|---|---|---|
    | **Same protocol** (OpenAI → OpenAI) | `__SAME_P50__` | `__SAME_P99__` | `__SAME_P999__` | `__SAME_CPU__` cores |
    | **Cross-protocol translate** (OpenAI → Anthropic, converted both ways) | `__XL_P50__` | `__XL_P99__` | `__XL_P999__` | `__XL_CPU__` cores |

    **The tail is flat — the no-GC proof.** Busbar is a single Rust binary with **no garbage collector**,
    so nothing in the request path pauses to sweep memory. Its added latency is near-constant: Busbar's
    own p99.9 is only **`__TAIL_RATIO__`× its p50** — the latency it adds does not grow a tail. A proxy on
    a garbage-collected runtime (Python/Node/JVM) pays an occasional GC pause that lands on *some*
    requests; those become a swollen p99/p99.9 even when the median looks fine. We report p50 / p99 /
    p99.9 precisely so the tail — where a no-GC architecture wins — is visible.

    **Throughput, so "under one core" isn't misread as a cap.** On its pinned cores Busbar sustained
    **`__RPS__` req/s**, i.e. **~`__RPS_PER_CORE__` req/s per core**, and the Tokio runtime scales across
    cores — this is efficiency, not a ceiling.

    ## Result 2 — vs the real Anthropic API: what that cost looks like in the wild

    Now we change *only the upstream* to `api.anthropic.com` and run the identical rig, cores, count, and
    concurrency. A real model call is **hundreds of milliseconds with a multi-second tail**, so:

    | Path (vs real Anthropic, `__MODEL__`) | p50 | p99 | p99.9 |
    |---|---|---|---|
    | **Without Busbar** (client → Anthropic) | `__A_DIR_P50__` | `__A_DIR_P99__` | `__A_DIR_P999__` |
    | **With Busbar** (client → Busbar → Anthropic) | `__A_BB_P50__` | `__A_BB_P99__` | `__A_BB_P999__` |

    The two rows are **statistically indistinguishable** — Busbar's `__SAME_P50__` sits far below the
    provider's own run-to-run variance. Against a real model call, **Busbar is invisible**: the overhead
    Result 1 measures in a clean room is, in production, below the noise floor of the thing it fronts. We
    show both rows so you can check the subtraction; we deliberately do **not** publish a p99.9 *delta*
    here, because at the extreme tail it is the difference of two provider-jitter tails — provider noise,
    not Busbar.

    :::note[Two true statements, one rig]
    **In a clean room, Busbar adds ~`__SAME_P50__` (and full cross-protocol translation only ~`__XL_DELTA_EXTRA__` more).
    Against a real provider, you cannot measure it at all.** Both come from the same machine, same cores,
    same load — only the upstream changed.
    :::

  <TabItem label="Reproduce">
    Every number above comes from one script on one named instance. Nothing is hand-tuned per result —
    the upstream is the only variable.

    **1. Launch the rig** — `c7g.2xlarge` (Graviton3, 8 vCPU), Amazon Linux 2023 (arm64), `us-east-1`:

    ```bash
    AMI=$(aws ssm get-parameter --region us-east-1 \
      --name /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-arm64 \
      --query 'Parameter.Value' --output text)
    aws ec2 run-instances --region us-east-1 --image-id "$AMI" --instance-type c7g.2xlarge \
      --key-name YOUR_KEY --security-group-ids YOUR_SG_ALLOWING_SSH
    ```

    **2. On the box** — install the released binary + `oha`, and a tiny low-jitter mock; pin the cores:

    ```bash
    curl -fsSL https://getbusbar.com/install.sh | sh          # released aarch64 busbar (rc.5)
    sudo dnf install -y golang                                # for the mock
    # oha (Rust load-gen): prebuilt aarch64 binary
    # mock: ~30-line Go HTTPS server returning a fixed OpenAI / Anthropic body
    # cert: a real Let's Encrypt cert for a loopback host (e.g. 127-0-0-1.local-ip.sh -> 127.0.0.1),
    #       so the RELEASE binary (public-CA-only upstream TLS) trusts the local mock.
    taskset -c 0-3 ./busbar ... &     # Busbar -> 4 dedicated cores
    taskset -c 6,7  ./mock  ... &     # mock   -> its own cores
    # measure with oha pinned to the remaining cores:
    taskset -c 4,5 ./oha -n __N__ -c __C__ --output-format json ...
    ```

    **3. Subtract.** For each upstream (mock, then `api.anthropic.com`) run the baseline and the
    Busbar path and subtract per percentile. The full self-contained script (launch → pin → measure both
    upstreams → log CPU → tear down) is published with the release.

    :::note[Why these choices]
    - **A fixed cloud instance, not a laptop** — pinning the instance type, region, OS image, and binary
      is what makes the run reproducible by anyone. "I used my MacBook" is not a citable number.
    - **Pinned cores + logged CPU** — proves Busbar's number is the cost of *one core of work*, not the
      result of throwing a big box at it.
    - **An instant mock** for Result 1 — the only way to isolate Busbar's own cost from provider jitter;
      the real-Anthropic run (Result 2) is the companion that shows it in practice.
    - **Public-CA loopback cert** — the release binary trusts only public roots upstream (a deliberate
      SSRF stance), so the mock is served over a real Let's Encrypt cert on a loopback-resolving host.
    :::
  </TabItem>
</Tabs>

**Honest competitive note.** No apples-to-apples third-party figure exists to cite, because nobody
publishes one. **LiteLLM** (Python/FastAPI) adds overhead in the **millisecond** range with a **GC tail**
by construction, but ships no reproducible self-host overhead benchmark. **OpenRouter** is a **SaaS hop** —
its "overhead" is a public-internet round-trip, not a proxy cost, and there is no self-host to measure. A
reproducible self-hosted overhead number is uniquely Busbar's; we would rather ship the harness with an
honest method than a number nobody can check.
